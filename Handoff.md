# Satellite Portfolio AI Engine: Handoff Document

This document summarizes the current state of the Autonomous Satellite Portfolio AI Engine, detailing the active architecture, technical decisions, and a roadmap for the next development phase.

## Current Architecture

The system operates as a headless, read-only Python application designed to be triggered via a weekly cron job.

1. **Orchestration (LangGraph):** The workflow is managed by a deterministic Directed Acyclic Graph (DAG) state machine. State is passed securely between functional nodes: `node_ingest_data` -> `node_analyst` -> `node_critic` -> `node_valuation` -> `node_controller`.
2. **Portfolio Ingestion (IBKR Flex Web Service):** Instead of relying on brittle or incomplete third-party wrappers (e.g., `py-ibkr`), we use native Python `requests` and the standard `csv` module to pull data directly from the IBKR Flex Web Service.
   * **Rate-Limit Resilience:** An `.ibkr_cache.csv` local cache intercepts aggressive IBKR rate-limits (`ErrorCode 1001`), ensuring the pipeline never crashes during unexpected polling timeouts.
   * **Data Validation:** Parsed CSV rows are strictly coerced into a `PortfolioItem` Pydantic model for type safety.
3. **Market Data (yfinance):** Basic price discovery and core fundamentals are handled via `yfinance` to maximize free-tier coverage (especially for global ETFs and commodities).
4. **Agentic Logic (Pydantic AI):** LLM interactions (powered by Google Gemini via the `AGENT_MODEL` env variable) are heavily restricted using Pydantic AI schemas to prevent formatting hallucinations.
5. **Delivery:** The final output is dispatched via Python's native `smtplib` using an HTML template built with classic `<table>` structures to guarantee layout compatibility across strict email clients like Gmail.

## Completed Tasks

* **Native IBKR Client:** Fully deprecated `py-ibkr` in favor of a `requests` + `csv` implementation tailored specifically for the `OpenPositions` section of Flex Queries.
* **Commodity Interception:** Implemented logic to intercept "spot" and "miner" commodity ETFs (`GLDM`, `PSLV`, `COPX`, `URA`) which IBKR natively classifies as equities (`STK`), remapping them to `CMDTY` so they correctly hit macro-valuation logic rather than flawed DCF pipelines.
* **Email UI/UX Fix:** Refactored the email generation template away from modern CSS Grid/Flexbox into robust `<table>` layouts.
* **Test Modernization:** Updated the entire test suite to comply with Pydantic AI v1.97+ (`result.output`), utilizing robust `@patch` mechanics to isolate network requests (e.g., mocking IBKR and Yahoo Finance).
* **Git Synchronization:** All current changes have been committed and pushed to `origin/main`.

  * *Action Item:* Hook up the Financial Modeling Prep (FMP) API or deeper `yfinance` methods to inject live fundamental data into the DCF Monte Carlo simulation.

* **CRON Scheduling:** The system is fundamentally complete for its local scope; it must now be deployed to a VPS or scheduled via Windows Task Scheduler/Cron to execute `main.py` automatically.

## Current Assumptions

* **IBKR Configuration:** The user's IBKR Flex Query delivery configuration will permanently remain set to export `CSV` format, not `XML`.
* **Model Context Window:** The `gemini-2.5-flash` model can easily handle the context lengths generated by scraping historical transcripts and macro reports for 20+ assets in sequence.
* **Read-Only Scope:** The system will never attempt to execute trades; it is strictly an analysis and reporting tool.

## Strict Rules & Engineering Best Practices

1. **No Third-Party Bloat:** We favor Python Standard Library solutions (`csv`, `smtplib`, `requests`) over obscure or unmaintained dependencies (e.g., `py-ibkr`), provided we enforce strict boundaries using Pydantic models.
2. **Decoupled Math:** LLMs are explicitly forbidden from calculating intrinsic values directly. Agents extract sentiment and macro regimes, which are then fed into deterministic Python math functions (Monte Carlo distributions).
3. **Clean Code & Single Responsibility:** All functions and classes must do exactly one thing. Dependencies are injected rather than hardcoded.
4. **TDD and Boundary Mocking:** The test suite uses explicit `Arrange, Act, Assert` blocks. Tests must never mock the system under test—only external I/O boundaries (e.g., API calls, file systems).
5. **No Credentials in Source:** Adherence to the "Rule of 2" — all secrets must reside in `.env.local` and be loaded safely at runtime.
